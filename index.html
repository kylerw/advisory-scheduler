Advisory Scheduler

Assigns every student to an advisory led by a teacher already on their schedule. Everything runs in your browser — the file never leaves this computer.

How to prepare the spreadsheet (start here each year)

Export one file from the SIS with one row per student to be placed. Header rows at the top are fine — the tool detects and skips them automatically.

ColumnContents
AStudent ID number
BScheduling priority: A, B, or C. This only sets processing order (A students tend to have the fewest class options) — it is not a student attribute.
C–JThe student's teacher codes, one column per period (8 periods), numeric codes only. Leave a period blank if the student has no class then — that student simply has fewer advisory options.
  • Each distinct teacher code in the file = one advisory section. Teachers who need to be combined must already be merged into a single shared code before export.
  • Students who are pre-placed by hand must be left out of the file, and their advisories don't appear in it. Every section in the file starts empty.
  • Include only IDs and codes — no names, grades, or other student information.
  • Save as .xlsx or .csv.

Settings: cap is the hard maximum per section (never exceeded); target is the size the balancer aims for. Defaults: cap 29, target 26.

What you get: a master list (student → section), per-section rosters (printable), and a flagged list naming every row that couldn't be placed and exactly why — including which sections are the bottleneck, so fixes happen in the data, not by guessing.

How the scheduling logic works (rules refresher)
  1. Who is placed where: every student goes to an advisory led by one of the teacher codes in their own row — never anyone else's. Any of a student's own teachers is considered equally good.
  2. Processing order: students with the fewest distinct teacher options go first (1-option students before 2-option, and so on — this is the general form of "partial schedules first"). Ties break by priority A → B → C, then by row order in the file. Order is about who picks first, not who gets placed — see rule 5.
  3. Seat choice: each student is placed in their eligible section with the most open seats below the target; seats between target and the cap are used only when no under-target option exists. The cap is never exceeded, ever.
  4. Automatic repair: when a student's sections are all full, the tool looks for a chain of swaps — move someone with other options out of a full section, ripple as far as needed — and applies it. This is why placement order can't strand anyone unnecessarily.
  5. The guarantee: if a full assignment is mathematically possible within the cap, the tool finds it. Anyone flagged genuinely cannot be placed without changing the data or raising the cap — and the flag names the bottleneck sections and how many students are locked to them.
  6. Balancing: after everyone is placed, students are shifted from over-target sections into under-target ones (always within their own teachers) until sections sit as evenly as the constraints allow.
  7. Repeatability: the same file with the same settings produces the identical result every run — re-running is always safe.
